TOSCA模型

# TOSCA 模型是什么

TOSCA 模型是一种专门用于澄清需求、问题的模型。

通过展开画布，可以让问题变得清晰起来：它也是是用户场景建模，5W1H模型的进阶方法。

TOSCA 由五部分组成：

  • Trouble，是哪个麻烦真实地摆在眼前？
  • Owner，这是谁面临并且需要解决的问题？
  • Success Criteria，成功的标准是什么样子的？
  • Constraints，问题的限制是什么？
  • Actors，哪些人与这个问题利益相关，他们的诉求是什么？

它来自于《像高手一样解决问题》 (opens new window)，督促我们解决问题的前提是定义问题。

# 可以解决哪些具体问题

  • 问题陈述
  • 解决问题
  • 分析问题

TOSCA 模型相比较5W1H模型，更能将问题清晰地陈述，方便解决问题。

# 为什么 TOSCA 模型能起作用

  • 原理是什么
  • 前提是什么

当我们把一个问题摊开到桌面上时，问题的多维度复杂性就可以一览无余了。

# 如何用 TOSCA 模型解决问题

  • 知识的可操作性（结构化Structured）
  • 后续行动计划（Plan）

# T-挡在眼前的麻烦是什么？

我们都知道，问题是“预期”与“结果”的差距。

所以我们需要知道，挡在眼前的麻烦究竟是什么？对应的事实又是什么？

这要求我们，必须要澄清麻烦的事实性：

  1. 我们需要收集客观事实，而不是含糊不清的主观表达，比如一些基于立场的抱怨，它们夹杂着巨大能量，但对解决问题毫无帮助。
  2. 不要让对事实的解读，混到定义中去，这会让我们迷失方向。
  3. 需要追问解决的时机：为什么是现在去解决？为什么之前没有行动？由此来找到之前被忽略的信息。

# O-这是谁的问题？

有了对麻烦的定义后，我们需要找到问题的所有者——这个人通常需要对其负责。

在问题定义的阶段，所有者具备对问题陈述的裁决权。只有他对当前问题的定义感到满意，我们才能真正着手解决这个问题。

# S-成功标准是什么样子的？

找到问题所有者有一个好处是，可以直接询问他：你想要什么？这个过程可能是在连续追问好几个“为什么”。但更好的问法是：你认为，成功是什么样子的？在开放性问题下，我们往往能得到问题所有者对解决问题的“关键立场”。

利用好这个信息，可以借助SMART法则推导出可明确量化的目标。

在定义可量化目标的问题上有两种学派：

  1. 放弃量化，认为可以通过包容性扩展来凝聚大家，完成目标。
  2. 使用量化，但在明显不能满足目标时及时调整目标。后面这种方法演化成了广为熟知的“OKR挑战目标法”。

# C-限制是什么？

在解决任何问题的过程中，我们能做到的事情都是有限定条件的。我们必须从问题所有者出发，去对当前问题的限制进行定义：

  1. 永远都存在对成功标准的约束，否则问题就不可能是问题了。
  2. 问题所有者的能力和资源会成为另一种约束。
  3. 解决问题的过程存在一定的约束，可能是权限、时间、预算等，这些限制的存在使得我们不能接触到希望接触的人。

对于第 3 种约束，必须说服问题所有者来松开一些条件，使得问题存在被解决的可能性。

# A-哪些人利益相关？

对于我们的建议，这些人可能支持，也可能捣乱，这是不可控的。需要分析他们与问题之间的利益关系，就成了解决问题的必要命题。

比如盗版不仅是因为正版费用过高，还有可能是绑定了专属的终端使得这些人厌恶盗版。分析出他们的利益和立场，有助于我们更好地解决问题。

一个好的问题定义，必须能满足这些问题：我们能否意识到问题存在的麻烦？是不是站在问题所有者的立场？有没有成功标准？是否考虑到了约束的存在？有没有考虑相关行动者？

  • 按这个知识去做了，结果怎么样（Result）
